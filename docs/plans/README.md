# docs/plans — 実装計画のテンプレートと運用契約

機能追加・挙動変更の実装計画は、このディレクトリに 1 機能 1 ファイルで置く。
このファイルがテンプレートとライフサイクルの正(AGENTS.md からここに誘導される)。

## ライフサイクル(`make lint` がドリフトゲートで強制)

このディレクトリ直下の計画ファイルは冒頭(タイトル直後)に必ず `Status:` 行を持つ:

- `Status: active` — 設計・レビュー中/実装中。
- `Status: shipped (PR #n)` — 出荷済み(括弧内の出所は必須 — 単一 PR でない旧プランは経緯を書く)。**「実装メモ」セクションに計画からの
  逸脱を必ず追記してから flip する**(PR 本文に書いた逸脱の再掲でよい —
  追加コストほぼゼロ)。本文は書いた当時のまま残す。
- **アーカイブ** — 場所がステータス: `archive/` へ移動し、既存の歴史的バナー
  (現在の正へのポインタ付き)を先頭に付ける。`archive/` 内のファイルに
  `Status:` 行は不要で、lint ゲートの対象はこのディレクトリ直下のみ。

出荷 PR が Status flip と実装メモ追記を行う。出荷済みプランを更新せず放置する
ことが、このゲートが防ぐ事故そのもの(実例: data-query プランは見出しの
パッケージ構成決定が出荷物と逆のまま 9 日間権威ある顔をしていた)。

## テンプレート(セクション順が契約 — 変わりやすい決定を先に)

```markdown
# <計画タイトル>

Status: active

## Goal(ユーザー確認済み)
一行のゴールと、ユーザーがどこで確認したか。未確認ならここで確認を取る。

## Options considered(ユーザー可視面に触れる計画では必須)
コマンド名・フラグ語彙・出力レイアウト等の taste が絡む決定は、代替案を
2 つ以上スケッチし、選定理由を書く。taste は見て初めて判断できる
(unknown known)— 出荷後の rename 波(#454-#457)が最高コストの教訓。

## Blind spots
明示的な blind-spot pass の結果: 考慮していなかったリスク・制約・
「知らないことを知らない」領域。なければ「なし」と書く(省略はしない)。

## 変更されやすい決定(レビューはここから)
データモデル、型インターフェース、フラグ、UX、パッケージ配置。
機械的な詳細より先に置く — 根本の不一致を安い段階で見つけるため。

## References
実装が従う既存コードの file:line(散文より正確な仕様として)。

## Open items / live-verify 台帳
未検証の仮定(応答形状・tenant 依存など)。解消先は「日付入り live-verified
コメント」か LIVE-UNVERIFIED マーカー(#521)— 計画ファイル内に放置しない。

## 機械的詳細
リファクタ手順・網羅リスト等。後回しで読める位置。

## Review log(append-only)
レビューの指摘と反映をパスごとに追記。過去のエントリは上書きせず、
覆った判断は「(第 N パスで訂正)」を付けて残す — 判断の履歴が価値。

## 実装メモ(出荷時に追記)
計画からの逸脱・理由・出荷後に判明した事実。Status: shipped と同時に必記。
```

## レビュー・プロトコル(多パス収束)

重要な計画は第二意見レビュー(Codex — 実行形態は AGENTS.md の Codex 節が正)を
**複数パス**回す。1 パスの CONVERGED を信用しない(実例: data-query プランは
パス 3 が CONVERGED を宣言した後、パス 4〜7 で実質的な P2 が出続け、
約 8 パスで真に収束した)。

- 毎パス**レンズを変える**。固定レンズ: ①前回反映の検証(verify-previous-fixes)
  ②新規発見(find-anything-new)③致命傷のみ(showstopper-only)。加えて
  **毎ラウンド 1 つ、即興のアドバーサリアル・レンズを必ず足す**
  (プロトコルのチェックリスト儀式化を防ぐガード)。
- 指摘は反映前に**一次ソース(コード・公式ドキュメント)で自分で検証**する。
  レビュアも間違える(自己訂正の実例あり)。
- 終了条件: **2 パス連続クリーン**。パスごとの指摘数の推移をユーザーに報告し、
  収束が見えるようにする。
